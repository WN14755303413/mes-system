# MES 项目管理系统 — 技术方案与实施计划

> 配套文档：`项目管理型MES系统建设方案.md`（业务方案）
> 本文档聚焦技术选型、安全设计、数据模型与分步实施路径。

---

## 1. 架构总览

采用业务方案第 12.1 节推荐的「快速落地型」架构：前后端分离的模块化单体，预留扩展接口。

```text
┌─────────────────────────────────────────────┐
│  浏览器 (React SPA)                          │
│  - 不持有任何密钥                             │
│  - 不直连数据库                               │
└──────────────────┬──────────────────────────┘
                   │ HTTPS + httpOnly Cookie
                   ▼
┌─────────────────────────────────────────────┐
│  NestJS 后端 (业务 + 鉴权 + 集成网关)          │
│  - 持有 DATABASE_URL / service_role key      │
│  - 所有权限校验在此发生                        │
└──────────────────┬──────────────────────────┘
                   │ 直连 Postgres (SSL)
                   ▼
┌─────────────────────────────────────────────┐
│  Supabase（复用现有项目）                      │
│  - Postgres  → schema: mes                   │
│  - Storage   → 图纸/附件/现场照片              │
└─────────────────────────────────────────────┘
                   ▲
                   │ 后续接入
        ┌──────────┴──────────┐
     用友 ERP              钉钉 / 蚂蚁分工
```

**关键决策：不使用 `supabase-js` 前端 SDK，不使用 Supabase Auth，不依赖 RLS 作为主要权限边界。**

理由：

1. `anon key` 必须打进前端包，F12 即可获取，违背用户的安全要求。
2. MES 的数据权限是「项目级 + 部门级 + 角色级 + 附件级」四维交叉（业务方案 §13），用 RLS 策略表达会极度复杂且难以审计。
3. 后端鉴权可以复用同一套 Guard，权限逻辑集中、可测试、可写审计日志。

Supabase 在此架构中的定位是**托管 Postgres + 托管对象存储**，即"基础设施即服务"，而非"后端即服务"。这是完全正当且常见的用法。

### 1.1 独立 Supabase 项目

**已决策：为 MES 新建一个独立的 Supabase 项目**，与现有管理数据库物理隔离。备份、扩容、故障域互不影响。

即便如此，仍在该项目内使用独立 schema `mes` 而非默认的 `public`：

```sql
CREATE SCHEMA IF NOT EXISTS mes;
-- 专用数据库角色，只授予 mes schema 权限
CREATE ROLE mes_app LOGIN PASSWORD '<from-env>';
GRANT USAGE ON SCHEMA mes TO mes_app;
GRANT ALL ON ALL TABLES IN SCHEMA mes TO mes_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA mes GRANT ALL ON TABLES TO mes_app;
-- 显式不授予 public / auth / storage schema 的写权限
```

这样即使后端连接串泄露，攻击者也接触不到 Supabase 自身的 `auth`、`storage` 元数据表。

Storage 侧建独立 bucket：`mes-drawings`（图纸）、`mes-attachments`（附件/现场照片），均设为 **private**，前端只能拿到后端签发的、有效期几分钟的预签名 URL。

## 3. 安全设计（重点）

用户核心要求：**账密不能通过 F12 破解出来。**

### 3.1 根本原则

> 前端不存在秘密。任何进入浏览器的东西都视为公开信息。

因此系统的安全边界完全落在后端。前端的角色隐藏菜单、禁用按钮，只是体验优化，**绝不作为访问控制手段**。

### 3.2 密码存储

- 明文密码只在两个瞬间存在：用户浏览器内存中、后端收到请求后的内存中。
- 入库前经 Argon2id 哈希（`memoryCost=19456, timeCost=2, parallelism=1`，OWASP 2024 推荐参数），带随机 salt。
- 数据库中只有 `password_hash` 字段。**无法反推明文**，管理员和 DBA 都看不到。
- 前端**不做**任何"预哈希"——那只会把哈希值变成事实上的密码，毫无意义。传输安全交给 HTTPS。

### 3.3 初始管理员账号

账号 `FAE@ADMIN` 通过一次性 seed 脚本创建：

```ts
// prisma/seed.ts —— 密码从环境变量读取，不出现在任何提交的文件里
const password = process.env.BOOTSTRAP_ADMIN_PASSWORD;
if (!password) throw new Error('BOOTSTRAP_ADMIN_PASSWORD 未设置，拒绝 seed');
await prisma.sysUser.create({
  data: {
    username: 'FAE@ADMIN',
    passwordHash: await argon2.hash(password),
    mustChangePassword: true,   // 首次登录强制改密
    // ...
  },
});
```

- `.env` 进 `.gitignore`，仓库只提交 `.env.example`。
- `mustChangePassword` 为 true 时，除「修改密码」外所有接口返回 `403 PASSWORD_CHANGE_REQUIRED`。
- 生产部署时 `BOOTSTRAP_ADMIN_PASSWORD` 由 Docker secret 或服务器环境变量注入，用完即可移除。

### 3.4 会话管理

```text
登录成功
  ├─ access_token  (JWT, 15 分钟)  → httpOnly; Secure; SameSite=Strict; Path=/
  └─ refresh_token (随机 256bit)   → httpOnly; Secure; SameSite=Strict; Path=/api/auth
                                      哈希后存 DB，支持吊销
```

- `httpOnly` 使 `document.cookie` 读不到 token，XSS 无法窃取，F12 的 Console 也拿不到值。
- Refresh token **轮换**（rotation）：每次刷新签发新 token 并作废旧的；检测到旧 token 复用 → 判定为窃取，吊销该用户全部会话。
- `SameSite=Strict` + 自定义 `X-CSRF-Token` 双重防护 CSRF。

### 3.5 登录防护

- 限流：单 IP 10 次/分钟，单账号 5 次失败后锁定 15 分钟（记入 `sys_login_attempt`）。
- 失败响应统一为「用户名或密码错误」，不区分账号是否存在，防用户名枚举。
- 恒定时间比较，失败路径也执行一次 dummy hash，防时序侧信道。
- 5 次失败后前端弹出图形验证码。

### 3.6 授权模型

后端 Guard 三层校验，逐层收窄：

1. **认证**：JWT 有效且未吊销。
2. **功能权限**：`@RequirePermission('project:create')` — 基于 RBAC 的权限点。
3. **数据权限**：`@DataScope('project')` — 拦截器改写查询条件。

数据权限规则（对应业务方案 §7）：

| 角色 | 可见范围 |
|---|---|
| 公司管理层 | 全部项目（只读 + 导出） |
| 项目经理 | 自己负责的项目全链路 |
| 部门负责人 | 本部门 + 下级部门 |
| 装配/调试人员 | 仅分派给本人的任务及关联图纸/工艺 |
| 系统管理员 | 全部（含配置与日志） |

### 3.7 审计与日志

`sys_audit_log` 记录：操作人、IP、UA、动作、目标对象类型/ID、变更前后 diff、时间戳。

图纸与 BOM 属核心资产，其**下载、预览、版本变更单独记录**（业务方案 §13）。

### 4.3 编码规则

严格遵循业务方案 §10.2，由后端 `CodeGeneratorService` 统一生成，序列号用 Postgres 序列保证并发安全。

### 4.4 通用状态机

`草稿 → 已发布 → 执行中 → (暂停 | 变更中) → 已完成 → 已关闭`，任意态可 `→ 已作废`。

状态流转由后端 `StateMachineService` 集中校验，非法跃迁直接拒绝。

## 5. 登录页设计

背景：`问题/1.png`（蓝色工业智造主题，右侧含数据看板与线框工厂）。

布局：

```text
┌────────────────────────────────────────────────┐
│  [背景图铺满，轻微 Ken Burns 缓慢缩放]            │
│  [顶层叠加深蓝渐变遮罩 + 流动光带 canvas]         │
│                                                │
│   MES 项目管理系统            ┌──────────────┐  │
│   半导体湿法装备制造执行平台    │  玻璃拟态卡片  │  │
│                              │  账号          │  │
│   ● 项目全过程管控            │  密码          │  │
│   ● 物料齐套穿透              │  [验证码]      │  │
│   ● 一机一档追溯              │  [登  录]      │  │
│                              └──────────────┘  │
│                                    v1.0 · 内部系统 │
└────────────────────────────────────────────────┘
```

动效（克制、不喧宾夺主）：

- 背景图 20s 循环 Ken Burns（scale 1.0 → 1.06），`prefers-reduced-motion` 时禁用。
- 卡片入场：Framer Motion，`opacity 0→1` + `y 24→0`，280ms `easeOut`。
- 输入框聚焦：底部渐变高亮线自中心向两端展开。
- 登录按钮：hover 时内部光泽扫过；提交时按钮宽度收缩为圆形 loading。
- 左侧标语逐条 stagger 淡入（延迟 80ms 递增）。
- 背景之上一层 `<canvas>` 绘制稀疏粒子与连线，随鼠标轻微视差位移，透明度 ≤ 0.25。

无第三方登录入口。密码框提供明文切换（眼睛图标），不提供"记住密码"（企业内部系统，避免凭据落盘）。

## 6. 分步实施计划

一次一个里程碑，每个里程碑结束时可运行、可验证。

| 里程碑 | 内容 | 交付验证标准 |
|---|---|---|
| **M0** | 仓库脚手架、Docker Compose、Supabase 连通、Prisma 初始化 | `docker compose up` 后前后端起来，后端能查通数据库 |
| **M1** | 认证与权限骨架：登录页、Argon2、JWT Cookie、限流、RBAC Guard、seed 管理员 | 用 `FAE@ADMIN` 登录成功；F12 拿不到 token；错密码 5 次锁定 |
| **M2** | 主框架：侧边导航、面包屑、页签、暗色/亮色主题、空状态与错误页 | 登录后进入首页，菜单按权限动态渲染 |
| **M3** | 系统管理：用户、角色、部门、权限点、审计日志查询 | 管理员可建角色并分配权限，越权请求被后端拒绝 |
| **M4** | 项目管理：台账 CRUD、里程碑、WBS、甘特图、风险与问题 | 建项目 → 编号自动生成 → 甘特图正确渲染 |
| **M5** | BOM 与图纸：版本管理、状态机、ECO、Storage 上传下载 | 草稿版 BOM 现场不可见；下载图纸产生审计记录 |
| **M6** | 物料齐套：需求分析、缺口计算、齐套看板 | 齐套公式（业务方案 §8.3）计算正确，标注同步时间 |
| **M7** | 生产计划与装配执行：计划、派工、报工、工时、异常上报 | 装配工只看到自己的任务；报工回写项目进度 |
| **M8** | 质量管理：检验单、质量问题单、8D 闭环、追溯 | 问题单从发现到关闭全流程走通 |
| **M9** | 调试与验收：调试记录、多轮整改复测、FAT/SAT 报告 | 生成 FAT 报告 PDF |
| **M10** | 数据看板：公司级、项目经理、齐套、质量、工时 | ECharts 看板加载 < 2s |
| **M11** | 集成预留与部署：ERP/钉钉适配器接口、接口日志、异常池、生产 Docker 编排 | 接口失败进异常池并可重试 |

一期范围对应业务方案 §14，即 M0–M8 + M10 的核心部分；M9、M11 视进度可并入二期。

## 8. 本地运行 → 生产 Docker 的一致性

- 本地和生产**共用同一份 Supabase**（或本地用 Supabase 的 dev 项目），避免"本地跑得通、线上跑不通"。
- 所有环境差异收敛到 `.env`，代码中零硬编码。
- 生产额外启用：nginx 反代 + TLS、`Secure` Cookie 强制、CSP 头、HSTS。
- 镜像多阶段构建，前端产物由 nginx 静态托管，后端跑 node:20-alpine。

## 9. 已确认的架构决策

1. **数据库接入**：Supabase 仅作 Postgres + 对象存储；业务后端自建 NestJS。前端零密钥。
2. **项目隔离**：为 MES 新建独立 Supabase 项目，内部再用 `mes` schema。

## 10. 待确认事项

1. 新 Supabase 项目的 `DATABASE_URL`（Session Pooler 连接串）与 `SERVICE_ROLE_KEY`，需填入本地 `.env`，**不要贴在聊天里**。
2. 是否已有用友 ERP 的接口文档 / 数据库只读账号？影响 M11 的适配器设计。
3. 钉钉企业内部应用的 AgentId / AppKey / AppSecret，用于后续免登与消息推送。
4. `FAE@123...` 作为初始密码强度不足，且已在明文渠道出现，建议部署前更换。
