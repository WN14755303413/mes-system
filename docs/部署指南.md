# 生产部署指南

> 适用范围：一期（M0–M11）单机 Docker 部署。数据库与对象存储使用 Supabase 托管，
> 服务器上只跑 `api`（NestJS）与 `web`（nginx + 前端静态资源）两个容器。

## 1. 架构与流量路径

```text
浏览器 ──HTTPS──> [TLS 终结] ──HTTP──> web 容器 (nginx :80，宿主映射 :8080)
                                        ├── 静态资源 /
                                        └── /api/ ──反代──> api 容器 (:3000，不对宿主暴露)
api ──直连 PG (Transaction Pooler)──> Supabase
```

- 浏览器不持有任何密钥、不直连数据库；`service_role key` 只存在于 api 容器的环境变量中。
- `COOKIE_SECURE=true` 时认证 Cookie 仅经 HTTPS 传输，因此 **TLS 是硬要求**，见 §4。

## 2. 服务器准备

- Linux x86_64，2C4G 起步（200 人规模日常够用），安装 Docker Engine ≥ 24 与 compose 插件。
- 开放 443（对外）；8080 仅供 TLS 反代回源，不要直接暴露公网。
- 服务器能访问 Supabase（出网 443/5432/6543）。

## 3. 环境变量清单（.env）

```bash
cp .env.example .env && chmod 600 .env
```

生产必查项：

| 变量 | 要求 |
|---|---|
| `NODE_ENV` | `production` |
| `DATABASE_URL` | Supabase Transaction Pooler（6543），带 `?schema=mes&pgbouncer=true&connection_limit=1` |
| `DIRECT_URL` | Supabase 直连（5432），迁移用，带 `?schema=mes` |
| `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` | 控制台 API 页获取；service_role 绝不下发前端/入库 |
| `STORAGE_DRIVER` | `supabase`（生产用对象存储；`local` 仅开发） |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | `openssl rand -base64 48` 各生成一个，两者不同 |
| `COOKIE_SECURE` | `true`（compose 已强制注入） |
| `WEB_ORIGIN` | 正式域名，如 `https://mes.example.com` |
| `BOOTSTRAP_ADMIN_PASSWORD` | 强密码；**seed 完成后从 .env 删除该行** |
| `ERP_ADAPTER` / `DINGTALK_ADAPTER` | 一期固定 `mock`；ERP/钉钉凭据留空即可 |

安全底线：

- `.env` 权限 600，绝不提交仓库、不贴聊天工具。
- 初始管理员首次登录会被强制改密（seed 内置），上线培训时明确告知。
- 一期集成为 mock 预留，`ERP_API_*`、`DINGTALK_*` 留空不影响运行；
  「系统管理 → 系统集成」页会如实显示凭据未配置。

## 4. TLS 终结（二选一）

`COOKIE_SECURE=true` 要求浏览器到系统的链路是 HTTPS，推荐在容器外做 TLS 终结。

### 方式 A：宿主机 Caddy（推荐，自动续期证书）

```text
# /etc/caddy/Caddyfile
mes.example.com {
    reverse_proxy 127.0.0.1:8080
}
```

### 方式 B：宿主机 nginx + certbot

```nginx
server {
    listen 443 ssl;
    server_name mes.example.com;
    ssl_certificate     /etc/letsencrypt/live/mes.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mes.example.com/privkey.pem;
    add_header Strict-Transport-Security "max-age=31536000" always;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

> 内层 web 容器的 nginx 已带 CSP、X-Frame-Options 等安全头与 `/api` 反代（`apps/web/nginx.conf`），
> 外层只负责 TLS，不要重复改写安全头。
> api 容器按「网段」信任反代（默认 `loopback,172.16.0.0/12`，可用 `TRUST_PROXY` 覆盖）：
> 解析 X-Forwarded-For 时跳过 docker 网桥与宿主机 nginx，审计日志与登录限流拿到的是客户端真实 IP。
> 若外层再加 CDN 等第三方代理，需把其回源网段加入 `TRUST_PROXY`，否则 IP 会停在 CDN 节点上。
>
> 关于审计日志里的 IP，两种属正常现象的「内网网关地址」：
> 本机开发时浏览器经 WSL/Docker Desktop 的 NAT 访问，源地址在进入应用前就已被改写为
> 网关（如 172.x.0.1），应用层无法还原（WSL 可改 `.wslconfig` 的 `networkingMode=mirrored` 解决）；
> 生产 Linux 服务器上不存在这一层 NAT，局域网用户会记录到真实 IP。

## 5. 首次部署

```bash
git clone <仓库地址> mes && cd mes
cp .env.example .env && vim .env      # 按 §3 填写
chmod 600 .env

docker compose up -d --build          # api 启动时自动执行 prisma migrate deploy

# 初始化权限点、内置角色与管理员（幂等，可重复执行）
docker compose exec api npx tsx prisma/seed.ts

# 验证
curl -fsS http://127.0.0.1:8080/api/health
```

浏览器访问 `https://mes.example.com`，用 `BOOTSTRAP_ADMIN_USERNAME` 登录，
按提示修改初始密码，然后从 `.env` 中删除 `BOOTSTRAP_ADMIN_PASSWORD` 行。

## 6. 版本升级

```bash
cd mes && git pull
docker compose build
docker compose up -d                  # 容器启动内置 migrate deploy，schema 随代码一起升级
docker image prune -f
```

如 shared 权限点有新增（如 M11 的 `sys:integration:write`），重跑一次 seed 即可补齐：

```bash
docker compose exec api npx tsx prisma/seed.ts
```

## 7. 数据备份与恢复

- 数据库：Supabase 免费档有 7 天自动备份；生产建议 Pro 档并周期性 `pg_dump`（走 `DIRECT_URL`）异地留存。
- 对象存储（图纸/现场照片）：Supabase Storage bucket 建议用 `rclone` 或控制台导出做周期备份。
- `.env` 与 TLS 证书：纳入服务器级备份，权限受控。

## 8. 日常运维

```bash
docker compose ps                     # 容器与健康检查状态
docker compose logs -f api           # 后端日志（含「MOCK」钉钉/ERP 模拟调用记录）
docker compose logs -f web
```

- 接口调用与失败补偿：登录系统 → 系统管理 → 系统集成 → 「接口日志与异常池」，
  失败记录可一键重试或标记人工处理（M11 验收标准）。
- 审计日志：系统管理 → 审计日志。
- 常见故障：
  - api 反复重启：看日志首屏的环境变量校验报错（zod 会列出缺失项）。
  - 登录报 Cookie 相关错误：确认访问走的是 HTTPS 且 `WEB_ORIGIN` 与实际域名一致。
  - 数据库连接数耗尽：确认 `DATABASE_URL` 走 6543 pooler 且带 `connection_limit=1`。

## 9. 二期接入真实 ERP / 钉钉时

1. 在 `.env` 填入 `ERP_API_*` / `DINGTALK_*` 凭据。
2. 后端在 `apps/api/src/modules/integration/adapters/` 新增真实适配器实现，
   并在 `integration.module.ts` 的工厂按 `ERP_ADAPTER` / `DINGTALK_ADAPTER` 取值切换。
3. 执行器、异常池、业务挂点、前端页面均无需改动——这是 M11「契约固化」的设计目标。
